On vmubtkube-a21 (Ubuntu 24.04, systemd 255), `NodeClockNotSynchronising` fired ~24h because `systemd-timesyncd` timed out against its only NTP server — OPNsense's LAN router advertised via IPv6 RA as a **link-local** address `fe80::29c:2ff:fea9:fdd0` (the LAN GUA/LL of igc1).

Root cause: the node is a Calico k8s worker with **20 interfaces each holding an equal-cost `fe80::/64 proto kernel metric 256` route** (physical NIC + 19 pod veths). An *unscoped* sendto to a link-local dest is resolved by the kernel against `main` (no per-iface policy tables for IPv6), and the tie-break hands it to a Calico veth (`cali…`) — so the NTP request never reaches the router. Proven by `ip -6 route get fe80::29c:2ff:fea9:fdd0 → dev cali…`; unscoped ping6 = 100% loss, scoped `-I enx…` = 0% loss; a zone-scoped NTP probe and the router's **GUA** both get replies in <1ms.

NOT a firewall issue: OPNsense pf passes LAN→router UDP/123 (line 86/87), ntpd listens on the exact `fe80…%igc1:123`, and the scoped probe succeeds — so the path is open; the packet just leaves the wrong interface.

Why no fallback saved it: timesyncd config is stock (`NTP=` empty). Per `timesyncd.conf(5)`, `FallbackNTP=` is only used when *no other* NTP info is known; the RA link-local server counts as "known", so `ntp.ubuntu.com` (reachable, 31ms v4) is never tried. The fix is to set `NTP=` to a non-link-local source (router GUA `2600:1700:76f0:14b0:29c:2ff:fea9:fdd0`, IPv4 `172.16.3.254`, or `ntp.ubuntu.com`) — timesyncd combines it with the link server and round-robins ("in turn, until one responds", systemd#33765). Most robust: chrony with explicit `pool`/`server`. Latent for the whole fleet — other worker nodes may trip it as pod/route counts shift; bake NTP config into provisioning (ansible key exists).

Confirmed on the node 2026-07-10: NTP arrives via **DHCPv6 stateless info-request**, NOT SLAAC/RA (RFC 8106 RDNSS is DNS-only). `/run/systemd/netif/links/228` (the LAN NIC) has `NTP=fe80::29c:2ff:fea9:fdd0` plus `DHCP6_CLIENT_DUID`/`DHCP6_CLIENT_IAID` — i.e. networkd's DHCPv6 client ran and pulled option 56. `timedatectl show-timesync` shows `LinkNTPServers=fe80::...`, `SystemNTPServers=` empty, and crucially `NTPMessage={ ... Ignored=yes, PacketCount=103 }`: timesyncd IS receiving NTP packets but rejecting them as non-matching (refines "never reaches router" → requests egress the wrong veth, so no validating reply; the 103 received packets are the router's unsolicited/broadcast NTP, origin-timestamp mismatch → Ignored).

How DHCPv6 runs with a static IP: netplan sets `dhcp4: no` + `accept-ra: yes` and **no `dhcp6:` key**, yet DHCPv6 still fires because the router's RA carries the Other/Managed flag and networkd auto-starts the DHCPv6 client under `IPv6AcceptRA`. So "NTP from DHCP with a static IP" works and is already in use here — DHCPv6 info-request delivers NTP without assigning an address.

Netplan source: `~/Documents/workspace/infra/k8s/k8s-setup-user-data.sh:65-89` writes `/etc/netplan/01-netcfg.yaml` during cloud-init. GOTCHA: that template names the NIC `enp1s0`, but vmubtkube-a21's live NIC is `enx00e04c6801c1` (USB ethernet) — so this node's netplan was produced/edited by a different path; verify before assuming template edits reach enx-named nodes.

Node-side fix = add `/etc/systemd/timesyncd.conf.d/10-ntp.conf` with `[Time]\nNTP=172.16.3.254 ntp.ubuntu.com` (SystemNTPServers merge with the broken link server; timesyncd round-robins to a working one). Netplan can drop the bad DHCPv6 NTP via `dhcp6-overrides: {use-ntp: false}`.

Online (systemd upstream): no dedicated "timesyncd won't scope link-local NTP" issue exists, but the identical bug class is fixed for **resolved** (#11935/#23655/#40534; PRs #12004/#17807 — pin socket via SO_BINDTOIFINDEX during connect(); IP_UNICAST_IF is a no-op on connected UDP sockets). timesyncd has no equivalent pinning. Closest timesyncd issues: #33224 (no per-iface NTP attribution), #29148 (DHCPv6 NTP handling, fixed v255), #20482 (IPv6 "Exchange full").

**RESOLVED 2026-07-10**, fleet-wide (both layers live on all 5 workers + codified in cloud-init). Two layers: (1) `/etc/systemd/timesyncd.conf.d/10-ntp.conf` → `NTP=172.16.3.254 ntp.ubuntu.com` — interface-independent, the one that actually carries the fix; the router serves NTP on its stable v4 so sync completes in <8s. (2) netplan `dhcp6: true` + `dhcp6-overrides: {use-ntp: false}` — suppresses the DHCPv6 link-local feed (`LinkNTPServers` → empty). Verified on every worker: `LinkNTPServers=` empty, `ServerAddress=172.16.3.254`, `System clock synchronized: yes`; all 8 k8s nodes `node_timex_sync_status=1`; no `NodeClockNotSynchronising` firing. Codified in `~/Documents/workspace/infra/k8s/k8s-setup-user-data.sh` for future provisioning. **GOTCHA — live NIC ≠ template's `enp1s0`:** layer-2 netplan patch anchors on the 5-space-indented `accept-ra: yes` line (NIC-name-independent) and was applied per-node to the real data NIC: a21=`enx00e04c6801c1`, a20=`enx00e04c680237`, a22=`enx00e04c680066`, a23=`enx00e04c68022c`, a24=`eno1` (onboard, not USB). a23 also has a dormant `enp1s0` stanza with `accept-ra: no` at 6-space indent — the 5-space anchor correctly skips it (only the data NIC got `dhcp6`+overrides). Backups at `/etc/netplan/01-netcfg.yaml.bak.<epoch>`. The template's hardcoded `enp1s0` still won't match these live NICs on a fresh cloud-init run — that template mismatch is a separate latent provisioning issue.
